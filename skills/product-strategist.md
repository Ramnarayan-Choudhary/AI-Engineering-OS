# Skill: Product Strategist

## Purpose

Decide what's worth building and why, before effort is spent building it — anchored in user value, not features.

## When to use

- Framing a new project, feature, or MVP.
- Deciding scope, or whether to build something at all.
- When "build X" deserves the question "should we, and what's the smallest version that proves it?"

## Inputs needed

- Who the user is and the problem they actually have.
- What success would look like (the observable outcome).
- Constraints: time, effort, what already exists.

## Process

1. State the user and their real problem in one sentence — resist jumping to the solution.
2. Define the value: what changes for the user if this exists.
3. Define success criteria you could actually measure or observe.
4. Find the smallest version that tests the core assumption (the MVP), and name that assumption.
5. Cut scope aggressively: what can be dropped, faked, or deferred without invalidating the test.
6. Recommend a scope and sequence, with what to build first and what to explicitly not build yet.

## Output format

```markdown
## User & problem
## Value proposition
## Success criteria
## Riskiest assumption
## MVP scope
  - In / Out / Deferred
## Recommended first step
```

## Success criteria

- The problem is stated in user terms, not solution terms.
- Success is observable, not vibes.
- Scope is minimal and tied to testing the riskiest assumption.

## Common mistakes

- Starting from the feature instead of the problem.
- MVP that's actually a maximal product.
- Success criteria that can't be observed.
- Building the safe part first and deferring the risky assumption that determines viability.

## When NOT to use

- What to build is already decided — move to `architect.md`/`planner.md`.
- It's a technical structure question, not a value question — use `architect.md`.
